# API Fetch Manager — SPEC tổng thể (System / Technical Spec)
> Tài liệu đặc tả kỹ thuật tổng thể. Mọi agent/dev đọc tài liệu này phải triển khai được đúng yêu cầu mà không cần hỏi thêm. Thuật ngữ kỹ thuật giữ nguyên tiếng Anh.

## 1\. Mục tiêu & Scope
### 1.1 Mục tiêu
Xây dựng hệ thống quản lý tập trung **API key của nhiều dịch vụ** (GitHub, Supabase, Cloudflare, Tailscale, dpdns...), gắn theo từng `emailOwner`, cho phép **thực thi API call** bằng key đã lưu, lưu **lịch sử + log chi tiết**, hỗ trợ **mở rộng API kiểu Fetch Builder**, và quản lý **bug/issue** trực tiếp từ UI qua chế độ Inspect.
### 1.2 In-scope
*   Quản lý credential/key theo `emailOwner` (CRUD, import/export)
*   Thực thi API call bằng key đã lưu
*   Fetch Builder (curl → endpoint tái sử dụng)
*   Lịch sử gọi API + logging chi tiết
*   Inspect Element Mode → tạo/quản lý bug/issue → export Markdown
*   Quản lý issue code (bug, feature)
### 1.3 Out-of-scope (v1)
*   Multi-tenant SaaS công khai
*   Billing/subscription
*   SSO doanh nghiệp

## 2\. Kiến trúc tổng thể (Monolith trong Docker)

```plain
┌─────────────────────────────────────────────┐
│                 Docker Container              │
│                                               │
│   ┌────────────┐        ┌──────────────────┐  │
│   │  Frontend  │  HTTP  │   Backend API    │  │
│   │ React + TS │ ─────► │  Node.js + TS    │  │
│   └────────────┘        │  (proxy + logic) │  │
│                         └────────┬─────────┘  │
│                                  │ Admin SDK   │
└──────────────────────────────────┼────────────┘
                                    ▼
              ┌──────────────────────────────────┐
              │      Google RTDB (multi-DB)       │
              │  • rtdb-keys   • rtdb-history      │
              │  • rtdb-logs   • rtdb-issues       │
              └──────────────────────────────────┘
```

*   **FE**: React + TypeScript, build tĩnh, phục vụ bởi BE.
*   **BE**: Node.js + TypeScript. Đóng vai trò **proxy** giữa FE và RTDB + nơi thực thi API call. Không cho FE chạm RTDB trực tiếp.
*   **Storage**: Google Realtime Database, tách nhiều DB độc lập.
*   **Deploy**: 1 Docker image, chạy self-host / GitHub Actions runner (Linux) / Azure runner (Linux).
### 2.1 Stack

| Layer | Công nghệ |

| Backend | Node.js + TypeScript (Express/Fastify) |

## 3\. Data Model RTDB
RTDB không có index kiểu SQL. Tối ưu truy vấn bằng: (a) **denormalization**, (b) khai báo `.indexOn` trong Realtime Database Rules, (c) cấu trúc key phẳng theo `emailOwner`.
### 3.1 rtdb-keys (quản lý credential)

```json
{
  "owners": {
    "<emailOwnerId>": {
      "email": "o861.clickup@gmail.com",
      "isSaveRtdbEmail": true,
      "createdAt": 1720000000000,
      "updatedAt": 1720000000000
    }
  },
  "credentials": {
    "<emailOwnerId>": {
      "<credId>": {
        "key": "github.token",
        "valueEnc": "<AES-GCM ciphertext>",
        "iv": "<base64 iv>",
        "service": "github.com",
        "label": "token chính",
        "createdAt": 1720000000000
      }
    }
  }
}
```

*   **1 key có thể có nhiều giá trị** → mỗi giá trị là 1 `credId` riêng, cùng field `key`.
*   Giá trị credential **luôn mã hoá** (`valueEnc` + `iv`). Không bao giờ lưu plaintext.
*   Rules `.indexOn: ["key", "service"]` trên node credentials.
### 3.2 rtdb-history (lịch sử gọi API)

```json
{
  "history": {
    "<emailOwnerId>": {
      "<historyId>": {
        "service": "github.com",
        "endpointId": "<fetchTemplateId>",
        "method": "POST",
        "url": "https://api.github.com/user/repos",
        "requestSummary": { "headers": {}, "bodyPreview": "..." },
        "responseStatus": 201,
        "responseSummary": "...",
        "durationMs": 342,
        "success": true,
        "calledAt": 1720000000000
      }
    }
  }
}
```

*   `.indexOn: ["service", "calledAt", "success"]`.
### 3.3 rtdb-logs (log chi tiết cho agent debug)

```json
{
  "logs": {
    "<logId>": {
      "level": "error",
      "scope": "fetch-executor",
      "service": "github.com",
      "business": "create-repo",
      "message": "401 Unauthorized",
      "detail": { "stack": "...", "request": "...", "response": "..." },
      "historyRef": "<historyId>",
      "createdAt": 1720000000000
    }
  }
}
```

*   Log phân theo `scope`, `service`, `business` để agent lọc nhanh. `.indexOn: ["service", "business", "level", "createdAt"]`.
### 3.4 rtdb-issues (bug/issue từ Inspect Mode + issue code)

```json
{
  "issues": {
    "<issueId>": {
      "type": "bug",
      "title": "Button lưu không phản hồi",
      "description": "...",

## 4\. Module / API Design (Backend)
Tất cả endpoint prefix `/api`. Response chuẩn: `{ ok: boolean, data?, error? }`.
### 4.1 Owner & Credential

| Method | Endpoint | Mô tả |
| ---| ---| --- |
| GET | `/api/owners` | Danh sách emailOwner |
| POST | `/api/owners` | Tạo owner |
| GET | `/api/owners/:id/credentials` | Danh sách key (giá trị trả về masked) |
| POST | `/api/owners/:id/credentials` | Thêm key (server mã hoá trước khi lưu) |
| PUT | `/api/credentials/:credId` | Sửa |
| DELETE | `/api/credentials/:credId` | Xóa |
| POST | `/api/import` | Import dữ liệu (JSON) |
| GET | `/api/export` | Export dữ liệu |

### 4.2 Fetch Execution

| Method | Endpoint | Mô tả |
| ---| ---| --- |
| POST | `/api/fetch/execute` | Body: `{ ownerId, templateId, params }`. Server lấy key (giải mã in-memory), build request, gọi, lưu history + log, trả kết quả. |
| POST | `/api/fetch/test-step` | Body: `{ ownerId, template, stepIndex, params }`. Chạy thử tới step đang chọn, trả response raw/JSON để viết extract; response vẫn redact credential đã lưu. |

### 4.3 Fetch Builder

| Method | Endpoint | Mô tả |
| ---| ---| --- |
| POST | `/api/templates/parse-curl` | Nhận chuỗi `curl`, parse thành template (method, url, headers, body, placeholder). |
| GET | `/api/templates` | Danh sách template |
| POST | `/api/templates` | Lưu template (kèm mapping output field cần giữ) |
| PUT/DELETE | `/api/templates/:id` | Sửa/xóa |

### 4.4 History & Logs

| Method | Endpoint | Mô tả |
| ---| ---| --- |
| GET | `/api/history?ownerId&service&success` | Lịch sử, filter |
| GET | `/api/logs?service&business&level` | Log, filter cho agent |

### 4.5 Issues

| Method | Endpoint | Mô tả |
| ---| ---| --- |
| GET/POST/PUT/DELETE | `/api/issues` | CRUD bug/issue |
| GET | `/api/issues/:id/markdown` | Trả nội dung Markdown để copy/export cho agent |

## 5\. Fetch Builder — Đặc tả chi tiết
**Mục tiêu:** thêm API mới mà không sửa code, bằng cách paste `curl`.

**Luồng:**
1. User paste lệnh `curl` đầy đủ.
2. Backend `parse-curl` → tách `method`, `url`, `headers`, `body`.
3. Hệ thống tự phát hiện phần cần thay bằng credential (vd `Authorization: Bearer <token>`) và cho user map tới `key` đã lưu → tạo **placeholder** `{{github.token}}`.
4. User khai báo **input params** (placeholder trong url/body) và **output mapping** (JSONPath các field cần lưu lại từ response).
   Có thể bấm **Test API** trên từng step để gọi thử với params runtime, xem response raw/JSON đã format và copy curl của step hiện tại.
5. Lưu thành **template** tái sử dụng.
6. Khi execute: server thay placeholder bằng credential (giải mã) + input params → gọi → trích output theo mapping → lưu history.

**Template schema:**

```json
{
  "id": "<templateId>",
  "name": "GitHub - Tạo repo",
  "service": "github.com",
  "business": "create-repo",
  "method": "POST",
  "urlTemplate": "https://api.github.com/user/repos",
  "headers": { "Authorization": "Bearer {{github.token}}" },
  "bodyTemplate": "{ \"name\": \"{{repoName}}\" }",
  "inputs": [ { "name": "repoName", "required": true } ],
  "credentialRefs": [ { "placeholder": "github.token", "key": "github.token" } ],
  "outputMapping": [ { "field": "repoUrl", "jsonPath": "$.html_url" } ]
}
```

## 6\. Inspect Element Mode — Đặc tả chi tiết
**Là 1 chế độ riêng**, bật/tắt bằng toggle.

**Luồng:**
1. Bật Inspect Mode → overlay trên UI.
2. User hover → highlight element; click → chọn (chọn được **nhiều element**).
3. Mở form nhập: `title`, `description`, `expectedResult`. Form tự đính kèm thông tin element đã chọn (`selector`, `outerHTML`, `boundingRect`).
4. Lưu vào `rtdb-issues` (có `status`, `createdAt`).
5. Có form quản lý (list + filter) để **sửa/xóa** từng bug/issue.
6. Nút **Export Markdown** / **Copy Markdown** để agent thực hiện theo.

**Markdown export mẫu:**

```markdown
## [BUG] Button lưu không phản hồi
**Status:** open · **Created:** 2026-07-07
**Description:** ...
**Expected:** ...
**Elements:**
- `#save-btn` — <button>Lưu</button>
```

## 7\. Security & .env
### 7.1 Nguyên tắc
*   Credential **mã hoá at-rest** bằng AES-256-GCM, khoá lấy từ env.
*   FE **không bao giờ** nhận plaintext credential; API trả về masked (`ghp_****7ANR1M`).
*   Backend giải mã credential **chỉ in-memory** khi execute, không log giá trị credential thật. Khi cần debug lỗi API đích có thể đặt `API_FETCH_MANAGER_REDACT_EXECUTION_VALUES=false` để không che các giá trị request/response thông thường; credential đã lưu vẫn luôn bị che.
*   Log tuyệt đối không ghi plaintext token.
### 7.2 Biến môi trường — bắt buộc prefix `API_FETCH_MANAGER_`

```plain
API_FETCH_MANAGER_PORT=8080
API_FETCH_MANAGER_ENCRYPTION_KEY=<32-byte base64>
API_FETCH_MANAGER_RTDB_KEYS_URL=https://xxx-keys.firebaseio.com
API_FETCH_MANAGER_RTDB_HISTORY_URL=https://xxx-history.firebaseio.com
API_FETCH_MANAGER_RTDB_LOGS_URL=https://xxx-logs.firebaseio.com
API_FETCH_MANAGER_RTDB_ISSUES_URL=https://xxx-issues.firebaseio.com
API_FETCH_MANAGER_FIREBASE_SA=<base64 service account JSON>
API_FETCH_MANAGER_LOG_LEVEL=info
API_FETCH_MANAGER_REDACT_EXECUTION_VALUES=true
```

## 10\. Bổ sung v1.1 — Fetch Flow, Extraction, Variables Store, Placeholder Engine
### 10.1 Fetch Template → Fetch Flow (sequential steps)
Template nâng cấp thành **flow gồm nhiều step**, chạy tuần tự, chia sẻ chung một `context`. Step sau tham chiếu output step trước qua `{{ctx.<stepId>.<field>}}`.

```json
{
  "id": "<flowId>",
  "name": "GitHub - Auth rồi tạo repo",
  "service": "github.com",
  "business": "create-repo",
  "stopOnError": true,
  "inputs": [
    { "name": "repoName", "required": true, "source": "runtime" },
    { "name": "orgToken", "source": "store", "varKey": "github.token" }
  ],
  "steps": [
    {
      "id": "auth",
      "method": "POST",
      "urlTemplate": "https://api.example.com/oauth/token",
      "headers": { "Content-Type": "application/json" },
      "bodyTemplate": "{ \"key\": \"{{orgToken}}\" }",
      "extract": [
        { "field": "accessToken", "jsonPath": "$.access_token" }
      ]
    },
    {
      "id": "createRepo",
      "method": "POST",
      "urlTemplate": "https://api.github.com/user/repos",
      "headers": { "Authorization": "Bearer {{ctx.auth.accessToken}}" },
      "bodyTemplate": "{ \"name\": \"{{repoName | lower}}\" }",
      "extract": [
        { "field": "repoUrl", "jsonPath": "$.html_url", "pinToVar": "github.lastRepoUrl" },
        { "field": "repoName", "jsonPath": "$.name" }
      ]
    }
  ]
}
```

**Executor logic:**
1. Khởi tạo `context` từ `inputs` (runtime → hỏi qua modal; store → lấy từ rtdb-variables; context → từ step trước).
2. Chạy từng step tuần tự: resolve placeholder → gọi request → `extract` output vào `context[stepId]`.
3. Field có `pinToVar` → ghi vào rtdb-variables để tái dùng.
4. `stopOnError: true` → dừng flow + log chi tiết khi step lỗi.
5. Lưu history cho từng step (kèm `flowId`, `stepId`).
### 10.2 Response Extraction (đa cấp, mảng, key)
*   Cú pháp: **JSONPath**. Hỗ trợ nested (`$.a.b.c`), mảng theo index (`$.items[0]`), wildcard map nhiều giá trị (`$.items[*].id`).
*   Mỗi extract: `{ field, jsonPath, pinToVar? }`. Nếu jsonPath trả mảng → lưu dạng array.
### 10.3 Input runtime
Mỗi input có `source`:

| source | Ý nghĩa |
| ---| --- |
| `runtime` | Hỏi người dùng lúc execute (render modal input) |
| `store` | Lấy từ kho biến (rtdb-variables) qua `varKey` |
| `context` | Lấy từ output step trước |

### 10.4 rtdb-variables (kho placeholder tái sử dụng)
DB thứ 5, tách riêng.

```json
{
  "variables": {
    "global": {
      "<varKey>": { "value": "...", "updatedAt": 1720000000000, "source": "manual|extracted" }
    },
    "<emailOwnerId>": {
      "github.lastRepoUrl": { "value": "https://github.com/...", "updatedAt": 1720000000000, "source": "extracted" }
    }
  }
}
```

*   Tham chiếu qua `{{var.<key>}}` (ưu tiên scope owner, fallback global).
*   Giá trị trích xuất (`pinToVar`) tự ghi vào đây. `.indexOn: ["updatedAt"]`.
*   **API bổ sung:** `GET/POST/PUT/DELETE /api/variables` (CRUD + filter theo scope).
### 10.5 Extracted Data API

| Method | Endpoint | Mô tả |
| ---| ---| --- |
| GET | `/api/extractions?ownerId&service` | Danh sách giá trị đã trích từ các lần fetch (repo name, git link, cloudflare NS id...), kèm nguồn template + thời điểm |

### 10.6 Placeholder Engine (transform + JS sandbox)
**Resolve order:** credential refs → variables → context → inputs → transforms.

**Cú pháp pipe:** `{{ source | transform1 | transform2(arg) }}`

**Transform whitelist (bắt buộc có sẵn):**
`upper`, `lower`, `trim`, `replace(a,b)`, `slice(a,b)`, `base64`, `base64decode`, `jsonStringify`, `urlEncode`, `date(format)`, `default(x)`.

**Advanced JS (tùy chọn, sandbox):**
*   Chạy trong **isolated VM** (`isolated-vm` hoặc tương đương).
*   **Cấm hoàn toàn**: network, filesystem, process, require/import.
*   Timeout cứng: 200ms. Memory limit. Chỉ nhận input là biến placeholder, trả về string.
*   Biểu thức có quyền đọc `ctx`, `inputs`, `vars` (read-only snapshot).

```plain
// ví dụ advanced expression\nreturn (inputs.repoName || 'repo').replace(/\\s+/g, '-').toLowerCase();\n```
